# Idea 扩展与延伸参考文档

本文档提供当原始想法存在根本性障碍时，如何系统性地扩展和延伸 idea 的详细方法论。

## 核心思路

不要局限于用户描述的具体应用场景，而是从中提炼出**底层模式**（核心交互范式 + 技术能力），然后将该模式迁移到更高频、更有商业潜力的场景中。

## 执行步骤

### 第一步：提炼底层模式

从用户的原始 idea 中抽象出核心的技术/交互模式，而非具体的应用场景。

**提炼时关注三个维度**：
- **输入方式**：用户如何提供信息（拍照、语音、视频、文字、扫码等）
- **AI 处理能力**：系统能自动做什么（识别、分类、索引、推荐、估值等）
- **输出价值**：用户最终获得什么（快速检索、决策辅助、自动化管理等）

**示例**：
- 原始 idea："拍照记录物品位置" → 底层模式："多模态 AI 自动录入 + 语义索引 + 模糊搜索"
- 原始 idea："语音记录会议内容" → 底层模式："语音实时转结构化数据 + 智能检索"
- 原始 idea："扫描食品标签追踪营养" → 底层模式："视觉识别 + 数据结构化 + 趋势追踪"

### 第二步：问题诊断表

明确原始 idea 的核心问题，输出结构化诊断：

| 问题维度 | 原始 idea 的表现 | 理想状态 |
|---------|-----------------|---------|
| 使用频率 | [如：低频，一年几次] | 高频（至少每周） |
| AI 准确度 | [如：物品种类杂，识别不稳定] | AI 能在该领域达到高准确率 |
| 付费意愿 | [如：痛点不够强，用户可忍受] | 用户有明确的付费动力 |
| 市场规模 | [如：小众需求] | 足够大的可服务市场 |

此表的作用是为后续替代方向提供明确的改善目标——每个替代 idea 都应在原始 idea 的弱项上有所提升。

### 第三步：场景迁移搜索

使用 `web_search` 工具，基于提炼出的底层模式，搜索该模式在其他场景中的应用：

**搜索策略**：
- 搜索底层技术能力在不同领域的应用（如 `"AI photo recognition [领域] app"`）
- 搜索相邻市场中用户痛点更强、频率更高的场景
- 搜索成功产品中使用类似技术模式的案例
- 搜索投资人/媒体对相关技术趋势的分析

**搜索关键词模板**：
```
# 技术模式迁移搜索
"[核心技术能力] app" [高频场景关键词]
"[核心技术能力]" startup idea 2025 2026
"AI [输入方式]" "[输出价值]" app market

# 相邻市场搜索
"[原始场景] alternative use case"
"[底层技术]" "high frequency" app idea
best "[相邻领域]" app AI powered

# 成功案例参考
"[类似技术模式的知名产品]" "similar to" OR "alternative"
"[类似技术模式]" startup funding raised
```

### 第四步：生成替代 idea

基于搜索结果，提出 2-4 个替代方向。每个方向须包含以下要素：

1. **一句话描述**：这个 idea 是什么
2. **与原始 idea 的关系**：保留了哪些底层模式，改变了什么
3. **为什么更好**：针对问题诊断表中的每个弱项，逐项说明替代方向如何改善
4. **市场验证信号**：搜索到的支持这个方向的证据（已有竞品、社区讨论、市场数据等）
5. **风险提示**：这个替代方向自身的主要风险

**替代方向的筛选标准**（优先推荐满足更多条件的方向）：
- 使用频率显著高于原始 idea
- AI 在该领域的识别准确率更高（目标领域的数据更标准化）
- 已有市场验证信号（竞品存在但有明显缺陷，或社区有强烈需求讨论）
- 与原始 idea 的技术栈高度复用
- 商业化路径更清晰（付费场景明确）

### 第五步：对比输出

用表格对比原始 idea 和所有替代方向，帮助用户做出直观决策：

| 维度 | 原始 idea | 替代 A | 替代 B | 替代 C |
|------|----------|--------|--------|--------|
| 使用频率 | ... | ... | ... | ... |
| AI 准确度 | ... | ... | ... | ... |
| 付费意愿 | ... | ... | ... | ... |
| 市场规模 | ... | ... | ... | ... |
| 技术复用度 | 基准 | ... | ... | ... |
| 竞争格局 | ... | ... | ... | ... |

### 第六步：给出推荐

- 明确推荐最看好的 1 个方向，并给出具体理由
- 理由应回扣问题诊断表，说明推荐方向如何系统性地解决原始 idea 的弱点
- 尊重用户选择，不强制推荐

## 输出后的流程

- **用户选择替代方向**：从阶段 1 重新开始（但可快速过渡，复用已有调研信息）
- **用户继续原方向**：尊重决定，继续推进后续阶段
- **用户希望融合多个方向**：协助整合出新的产品概念，再从阶段 1 重新评估
